# Issue 09 - P0: `axiom init` writes an empty project OpenRouter key that masks a valid global key

**Status:** Open  
**Severity:** P0  
**Date opened:** 2026-04-08  
**Source:** `Testing Results/Test 1/report.md`  
**Base commit:** `main` @ `9019c10`

---

## 1. Issue

A user can configure a valid OpenRouter API key in the global config at `~/.axiom/config.toml`, run `axiom init`, and still end up with `axiom status` / `axiom tui` failing as if no key exists.

The failure observed in Test 1 was:

```text
error: no inference provider available for configured orchestrator runtime: runtime "claw" requires an openrouter API key
```

The root cause was not a missing global key. The generated project config contained an explicit empty value:

```toml
[inference]
openrouter_api_key = ""
```

That empty project-level field overrode the valid global key during layered config loading.

---

## 2. Recreation

1. Put a valid non-empty `inference.openrouter_api_key` in `~/.axiom/config.toml`.
2. Run `axiom init --name "<project>"` in a clean repository.
3. Inspect `.axiom/config.toml`.
4. Observe that `internal/project/project.go:54-60` wrote a full default config into the project file.
5. Observe that the serialized default config includes an explicit `openrouter_api_key = ""` because `internal/config/config.go:137-178` marshals the zero-value field from `config.Default(...)`.
6. Run `axiom status` or `axiom tui`.
7. Observe that startup fails with the runtime/provider error because `internal/config/config.go:312-344` loads global config first, then project config, and `internal/config/config.go:348-369` makes any explicitly present project field authoritative.

---

## 3. Root Cause

Three parts combine to create the defect:

1. `config.Default(name, slug)` intentionally leaves `Inference.OpenRouterAPIKey` empty.
2. `project.Init(...)` writes the entire default config into `.axiom/config.toml` instead of writing only project-scoped settings.
3. `config.Load(...)` merges project config over global config using explicit field presence, not semantic meaning.

That merge behavior is reasonable for real project overrides. The bug is that `axiom init` manufactures an explicit empty override for a user-global secret field that should not have been written into the project config in the first place.

---

## 4. User Impact

- A correct global OpenRouter setup appears broken immediately after `axiom init`.
- `axiom status`, `axiom tui`, and any other startup path gated by the inference-plane check fail on a newly initialized project.
- The failure message points users at the key, but the actual problem is config layering, not missing credentials.
- The bug pushes users toward storing secrets in project-local config files to "fix" the symptom, which is the wrong security boundary.

This is a first-run blocker, so it is a P0 issue.

---

## 5. Plan To Fix

The fix should be applied at project-template generation, not by weakening config merge semantics.

Recommended fix:

1. Change `internal/project/project.go` so `axiom init` writes a project config template that includes only project-scoped settings.
2. Omit secret-bearing or user-machine-scoped fields from the generated project config by default, especially:
   - `inference.openrouter_api_key`
   - other future credential fields in `[inference]`
3. Keep layered loading behavior in `internal/config/config.go` as-is for real overrides that users intentionally place in project config.
4. Update docs to state clearly that OpenRouter credentials belong in `~/.axiom/config.toml`, not in `.axiom/config.toml`.

An acceptable alternative would be to introduce an explicit "inherit" representation for secret fields, but that is more invasive than simply not writing the field into the project config template.

## 7. Acceptance Criteria

- [ ] A newly initialized project does not contain `inference.openrouter_api_key` in `.axiom/config.toml` unless the user explicitly added it.
- [ ] With a valid global OpenRouter key and a freshly initialized project, `config.Load(projectRoot)` preserves the global key.
- [ ] `axiom status` and `axiom tui` start successfully after `axiom init` when the only configured key is global.
- [ ] No project template generated by `axiom init` encourages storing provider secrets in repo-local config.
