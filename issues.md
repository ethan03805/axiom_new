## Test-4 Issue 1 — Hook policy conflicts with the orchestrator's required actions

**Severity:** Blocker / Design tension

The `.claude/hooks/axiom-guard.py` generated by `axiom skill generate --runtime claude-code` blocks every `Edit`, `Write`, `MultiEdit`, and non-`axiom` `Bash` invocation. Several legitimate orchestrator actions need exactly those tools. Also: the hook only activates on the **next** Claude Code session, not the current one.

Suggested fixes:
- Add `axiom srs submit -` (stdin) so SRS can be piped.
- Add CLI verbs that wrap the control-WebSocket types (`axiom task create`, `axiom task approve`, etc.).
- Or ship an `axiom orchestrate` command that opens the control WebSocket itself.
- Document that the runtime hook is incompatible with sessions where the orchestrator must also produce auxiliary artifacts on disk, and print a warning from `axiom skill generate` that the hook only takes effect on the next session.

## Test-4 Issue 2 — 🚨 CRITICAL: `GET /api/v1/projects/:id/status` leaks the OpenRouter API key in plaintext

**Severity:** CRITICAL — secret disclosure

The response body contains `ConfigSnapshot` which is the literal JSON of the entire config tree, including `Inference.OpenRouterAPIKey`. Anyone with a read-only scoped token can exfiltrate the user's OpenRouter key. This contradicts the "API key never appears in logs" guarantee in getting-started.md.

Suggested fixes:
1. Redact `ConfigSnapshot.Inference.OpenRouterAPIKey` (and any `*_api_key`, `*_secret`, `*_token`) at the response encoder.
2. Consider not returning `ConfigSnapshot` at all from `/status`.
3. Add a response-body test that fails if any `sk-*`, `axm_sk_*`, or configured-secret pattern appears in responses.
4. Rotate the leaked key (user action).

## GitHub Issue #1 — Regular text in bootstrap mode calls Engine.StartRun directly

**Severity:** High / Design

(a) **Bootstrap-mode TUI clobbers an existing run.** Starting `Engine.StartRun` should fail (or prompt) when a run already exists in `draft_srs`/`awaiting_srs_approval`/`active`/`paused`. Today, a typo replaces in-flight orchestrator work with zero confirmation.

(b) **Slash-command affordance isn't discoverable.** The TUI accepts `srs show` as a prompt and `/srs show` as a command. A user who's been typing `axiom srs show` on the CLI will naturally drop the slash. The TUI should reject bare shell-style commands in bootstrap mode, or at minimum warn "did you mean /srs show?".

(c) **No audit trail surfaced.** The export shows only the current run. Orphaned prior runs don't appear anywhere the user can see, even though their draft files are still on disk.

## GitHub Issue #2 — axiom init should automatically initialize git

**Severity:** Medium / UX

User has to manually initialize git prior to starting a task. `axiom init` should `git init` automatically if the working directory is not already a git repo.

### 1. P0: `axiom run` dead-ends at `draft_srs`, and the user prompt is ignored

- A first-time user can start a run, but the system does not generate an SRS, does not persist a draft, and does not expose a normal user path to continue.
- The core product promise in the architecture and quick-start docs is "describe what to build, review the SRS, then execute." The current runtime stops at step 1.

Likely root cause:

- The architecture assumes an orchestrator/bootstrap layer, but the composition root never wires an embedded orchestrator or an external-orchestrator handoff that actually consumes the run prompt and calls the SRS lifecycle.

Docs affected:

- `docs/getting-started.md`
